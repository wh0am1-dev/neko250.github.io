patrones de diseño

patrón Singleton

  • patrón de creación de objetos
  • único ejemplar de una clase
  • mecanismo de acceso al ejemplar

implementación sin patrón

  • se puede usar una variable global:
public static Contador contador = new Contador();
// ...
Contador.contador.nuevoIdentificador();
  • pero esto no evita que se puedan crear varios ejemplares
miContador = new Contador();
  • o que se destruya
Contador.contador = null;

consideraciones

  • para asegurar un único ejemplar
    • constructor privado
  • el almacenamiento del ejemplar
    • variable estática privada
  • acceso al ejemplar
    • a través de un método estático público
  • creación del ejemplar
    • la primera vez que se invoque el método de acceso

solución

public class Singleton {
    private static Singleton ejemplar;
    
    private Singleton() {}
    
    public static Singleton ejemplar() {
        if (ejemplar == null)
            ejemplar = new Singleton();
        
        return ejemplar;
    }
    
    public void operacion() {} // adicional
}

patrón Método Fábrica

  • patrón de creación de objetos
  • sin indicar la clase concreta del objeto creado
    • no se invoca a un constructor sino a un método abstracto Fábrica
  • aplicación a la creación de objetos del patrón Plantilla de Métodos
    • define una interfaz para crear un objeto
    • deja su implementación a las subclases
  • se aplica cuando:
    • un sistema debe fabricar / crear objetos
    • debe ser independiente de los productos concretos fabricados

implementación

public abstract class MazeGame {
    public MazeGame() {
        Room room1 = makeRoom();
        Room room2 = makeRoom();
        room1.connect(room2);
        this.addRoom(room1);
        this.addRoom(room2);
    }
    
    abstract protected Room makeRoom(); // método fábrica
}
public class OrdinaryMazeGame extends MazeGame {
    protected Room makeRoom() {
        return new OrdinaryRoom();
    }
}
public class MagicMazeGame extends MazeGame {
    protected Room makeRoom() {
        return new MagicRoom();
    }
}

patrón Fábrica Abstracta

  • patrón de creación de objetos
  • proporciona una interfaz para crear familias de objetos relacionados
  • sin especificar las clases concretas de las que son instancia
  • se aplica cuando:
    • un sistema debe fabricar o crear objetos
    • debe ser independiente de los productos concretos fabricados
    • pueden fabricarse varias familias intercambiables de productos
    • los productos de una familia deben usarse conjuntamente, no pudiendo combinarse productos de familias distintas

problema

  • considerar diferentes frameworks de creación de GUIs
    • swing, motif, open windows...
  • cada framework define una serie de elementos similares pero definidos diferentemente en cada uno

solución

  • para cada componente se define una clase abstracta (o interfaz)
interface Ventana {
    // ...
}

interface Boton {
    // ...
}

interface Menu {
    // ...
}
  • definimos una clase abstracta (o interfaz) FábricaDeComponentes, que declara los métodos fábrica para crear cada tipo de componente
abstract Ventana crearVentana();
abstract Boton crearBoton();
abstract Menu crearMenu();
  • FábricaDeComponentes se especializa en distintas fábricas, una para cada estándar GUI
class FabricaMotif extends FabricaDeComponentes {}
class FabricaPM extends FabricaDeComponentes {}
class FabricaOW extends FabricaDeComponentes {}
  • cada componente abstracto se especializa para cada estándar
class VentanaMotif implements Ventana {}
class VentanaPM implements Ventana {}
class VentanaOW implements Ventana {}

consideraciones

  • los clientes quedan aislados de los componentes concretos
    • la responsabilidad de crear objetos se encapsula en las fábricas
  • facilita el intercambio de familias de productos
    • el nombre de una fábrica concreta aparece sólo una vez en la aplicación
  • la extensión para una nueva familia de productos es sencilla
    • crearemos nuevas clases fábrica concreta y productos concretos
  • la extensión para nuevos tipos de productos no es fácil
    • la interfaz de FábricaAbstracta fija los productos que pueden crearse
    • cuando existe un ascendente común a todos los productos, este inconveniente se puede resolver mediante casting
    • pero si la naturaleza de los nuevos objetos a crear es distinta, la reutilización se ve comprometida
  • normalmente sólo se necesita un ejemplar de cada fábrica
    • las fábricas concretas pueden utilizar el patrón Singleton

patrón Prototipo

  • patrón de creación de objetos
  • especifica el tipo de objetos que se deben crear utilizando un ejemplar como prototipo
  • crea nuevos objetos a partir del prototipo
  • se aplica cuando:
    • un sistema debe ser independiente de cómo se crean, componen y representan sus productos
    • las clases a instanciar se especifican en tiempo de ejecución (carga dinámica)
    • se quiere evitar construir jerarquías de clases paralelas
    • las instancias de una clase pueden contener una de un número limitado de combinaciones de estados

ejemplo

  • consideremos la construcción de un editor de pentagramas
    • se pretende reutilizar un marco de trabajo para editres gráficos mediante la incorporación de nuevos objetos que representan notas y otros elementos musicales
  • el editor de pentagramas dispondrá de una paleta de herramientas para añadir estos elementos musicales al pentagrama

  • la paleta también incluirá herramientas básicas para seleccionar, mover y manipular estos elementos, funcionalidad ofrecida por el marco de trabajo

  • supongamos que el marco proporciona:

    • una clase abstracta Graphic
      • con subclases para los componentes gráficos
    • una clase abstracta Tool
      • con subclases para las herramientas que constituyen la paleta
    • una subclase de Tool, CreationTool
      • para herramientas que crean instancias de objetos gráficos y las añaden al documento

  • la clase CreationTool presenta un problema

    • las clases para notas y demás elementos musicales son específicas de nuestra aplicación
    • pero CreationTool pertenece al marco genérico, y desconoce cómo crear instancias de las clases musicales

solución inicial

  • especializar CreationTool para cada objeto musical
  • produce una jerarquía de clases que reproduce la de Graphic y que sólo difieren en el tipo de objeto musical a instanciar

solución alternativa

  • parametrizar CreationTool con el Graphic que debe crear
  • CreationTool creará objetos clonando una instancia de la subclase adecuada de Graphic
  • las subclases de Graphic deben implementar la interfaz Clonable

patrón Adaptador

  • también se conoce como Wrapper
  • patrón estructural con una versión para clases y otra para objetos
  • se aplica cuando:
    • se desea reutilizar una clase existente pero su interfaz no concuerda con la que se necesita
  • adapta la interfaz de una clase a la interfaz esperada por sus clientes
  • favorece la reutilización (de la clase adaptada) y permite la colaboración entre interfaces incompatibles

ejemplo

    • por ello se desea reutilizar la clase Text de la biblioteca de clases del entorno de programación
  • la interfaz de Text no se corresponde con ObjetoGráfico
    • implementa operaciones edit y draw
  • se necesita desarrollar un adaptador
    • que adapte Text a la interfaz declarada por ObjetoGráfico

  • el tipo de adaptación puede ser muy variado
    • desde una simple conversión de interfaces hasta añadir un conjunto totalmente diferente de operaciones

patrón Representante

  • patrón estructural de objetos
  • proporciona un representante (proxy) para controlar el acceso a un objeto

primera aproximación

  • utilizamos un puntero nulo para hacer referencia a la imagen cuando no está aún cargada

  • inconvenientes:

    • un puntero no es un objeto, no puede recibir mensajes
    • se complica el editor, que debe conocer el uso del objeto y distinguir si está cargado o no
    • no permite almacenar datos necesarios como el tamaño de la imagen

solución

  • utilizaremos un objeto como representante de la imagen
  • el representante hará las funciones de la imagen mientras aún no esté cargada

  • el mecanismo básico utilizado es la delegación

funcionamiento

  • utilizamos un objeto como representante

  • el estado de la imagen (cargada o no) es transparente al editor

tipos de proxies

  • proxy remoto
    • proporciona un representante local a un objeto remoto, ocultando que cliente y representado residen en espacios de direcciones distintos
  • proxy virtual
    • permite crear objetos pesados bajo demanda
  • proxy de protección
    • controla el acceso al objeto representado
  • referencia inteligente
    • reemplaza a un puntero para permitir funciones adicionales

estructura

  • el proxy debe apuntar al objeto representado

  • las interfaces de representante y representado son sustancialmente similares

  • los objetos representados pueden ser de distintos tipos
  • el proxy permite un acceso transparente

patrón Estrategia

  • patrón de comportamiento de objetos
  • define una familia de algoritmos, los encapsula en clases distintas y los hace intercambiables
  • permite variar los algoritmos haciéndolos independientes de los clientes que los utilizan

ejemplo

  • existen múltiples algoritmos para dividir un párrafo en líneas
    • dependiendo del lenguaje en el que está escrito
  • no es deseable incluirlos todos dentro de las clases que los pueden requerir
    • las clases se hacen más complejas
      • más grandes y difíciles de mantener
    • cada algoritmo puede ser apropiado en un momento determinado
    • es difícil añadir nuevos algoritmos o modificar alguno existente

alternativa

  • definir clases estrategia por cada uno de los algoritmos

solución

 consideraciones

  • define una familia de comportamientos intercambiables
  • las estrategias son diferentes implementaciones del mismo comportamiento
  • permite organizar la funcionalidad común de las estrategias
  • evita utilizar construcciones condicionales para seleccionar el comportamiento deseado
  • la interfaz de Estrategia es compartida por todas las clases que determinan estrategias concretas
    • es probable que algunas estrategias no utilicen toda la información
  • incrementa el número de objetos en la aplicación
    • Singleton permite implementar las estrategias como objetos sin estado que los contextos pueden compartir

patrón Estado

  • patrón de comportamiento para objetos
  • se aplica cuando:
    • un objeto pasa por diferentes estados, típicamente representados por un tipo enumerado
    • su comportamiento depende del estado
    • sus operaciones se implementan mediante grandes instrucciones condicionales que dependen del estado
  • permite que un objeto cambie su comportamiento
    • cuando cambia su estado
    • tal como si el objeto cambiase de clase

ejemplo

  • supongamos una clase Buffer que representa un buffer de tamaño 1 con métodos put y get
  • la extracción de elementos de un buffer vacío y la inserción en un buffer lleno produce una excepción

implementación sin patrón

class Buffer {
    enum Estado {
        vacio,
        lleno
    };
    
    private Estado estado;
    private Elemento elemento;
    
    public Buffer() {
        estado = Estado.vacio;
    }
    
    public void put(Elemento elem) {
        if (estado == Estado.lleno) {
            throw new RuntimeException("Buffer lleno");
        }
        
        elemento = elem;
        estado = Estado.lleno;
    }
    
    public Elemento get() {
        if (estado == Estado.vacio) {
            throw new RuntimeException("Buffer vacio");
        }
        
        estado = Estado.vacio;
        return elemento;
    }
}

consideraciones

  • codificar el estado mediante un valor enumerado
    • da lugar a un análisis de casos en los métodos
    • código complejo
    • problemas de mantenimiento

alternativa

  • codificar el estado mediante una clase abstracta
    • con tantas subclases como estados
    • permite explotar el polimorfismo de datos
    • haciendo corresponder cada rama de comportamiento condicional con una subclase

solución

consideraciones

  • la interfaz del Contexto se delega en su Estado
    • excepto los métodos independientes del estado
void operacion() {
    estado.operacion();
}
  • en los métodos de los estados
    • es necesario acceder a atributos y/o métodos del contexto
      • mediante una asociación bidireccional
      • pasando el contexto como argumento
    • se realizan los cambios de estado
      • causa interdependencia entre los estados
      • podemos usar el patrón Fábrica
  • si los objetos Estado no tienen atributos
    • pueden ser compartidos por todos los objetos Contexto
    • podemos usar el patrón Singleton

patrón Plantilla de Método

  • patrón de comportamiento para clases
  • define el esqueleto de un algoritmo en una operación
    • retrasando la implementación de los pasos a las subclases
  • las subclases redefinen ciertos pasos de un algoritmo
    • sin cambiar la estructura global del mismo

motivación

  • supongamos que:
    • queremos extraer información de una fuente externa (fichero)
    • los datos son tratados y almacenados en una estructura interna
  • el problema surge en que la forma de tratamiento, puede variar de clases que aún no conocemos, o que pueden variar en el futuro
  • codificar un tratamiento inicial directamente en el algoritmo de lectura puede comprometer el diseño en el futuro

consideraciones

  • se aplica:
    • para implementar las partes invariantes de un algoritmo, dejando a las subclases implementar el comportamiento que puede variar
    • cuando el comportamiento común entre subclases debe ser factorizado y localizado en una clase común para evitar duplicación de código

patrón Iterador

  • proporciona una forma de acceder secuencialmente a los elementos de una colección sin mostrar la representación interna de la misma
  • permite definir diversos recorridos sobre una estructura admitiendo el uso simultáneo de la misma por parte de varios clientes

solución

  • cada cliente tendrá acceso a la colección mediante una instancia de la clase Iterador
    • dicha instancia será proporcionada por la colección a la que se quiera acceder
  • la clase que implemente el iterador deberá tener acceso a la implementación de la colección sobre la que itera
  • se podrán proporcionar tantos iteradores como se desee, implementando diversos recorridos:
    • del principio al final (o viceversa)
    • según el orden de los elementos almacenados
    • de forma aleatoria

patrón Cadena de Mando

  • patrón de comportamiento de objetos
  • disminuye el acoplamiento entre el emisor de una petición y el receptor que finalmente la atiende
  • se consigue dando a más de un objeto la oportunidad de responder a la petición
  • los objetos receptores se encadenan y se van pasando la petición hasta que uno de ellos la atiende

motivación

  • consideremos un sistema de ayudas sensible al contexto en una interfaz gráfica de usuario:
    • el usuario puede obtener información de ayuda sobre cualquier parte de la interfaz situándose sobre ella
    • la ayuda que se obtiene depende de la parte de la interfaz sobre la que se solicita y su contexto
    • si no existe ayuda específica sobre el elemento del que se solicita, el sistema debería ofrecer un mensaje de ayuda más general sobre el contexto inmediato
  • la información de ayuda se organiza atendiendo a su profundidad en el contexto: de los más internos a los más externos
  • el problema radica en que el objeto que finalmente proporciona la ayuda no es conocido explícitamente por el objeto solicitante
  • la idea del patrón es desacoplar los emisores de los receptores dando a diferentes objetos la posibilidad de atender peticiones
  • la petición se va pasando de un objeto a otro a lo largo de una cadena "de mando" hasta que uno de ellos la atiende

 aplicación

consideraciones

  • este patrón libera a un objeto de conocer quién atiende sus peticiones: el objeto sólo tiene que confiar en que la petición se tratará adecuadamente
  • sin embargo, nada lo garantiza: si la peticiónno se atiende al final de la cadena, o ésta no está bien configurada
  • ni el emisor ni el receptor tienen conocimiento explícito uno de otro: en vez de mantener referencias a todos los receptores, el emisor sólo mantiene una referencia a su sucesor
  • habitualmente, en los casos susceptibles de aplicar este patrón, los enlaces que establecen la cadena de mando ya existen: patrón Compuesto o Decorador

patrón Mediador

  • patrón de comportamiento para objetos
  • define un objeto que encapsula cómo un conunto de objetos interactúa entre sí
  • el DOO promueve la distribución del comportamiento
    • pero esa distribución puede implicar estructuras con muchas conexiones entre los objetos
    • en el peor de los casos, cada objeto tiene conocimiento de todos los demás
  • dividir un sistema en objetos aumenta la reutilización
    • pero la proliferación de interconexiones tiende a disminuirla
    • los objetos son difícilmente reutilizables sin la presencia de los otros con los que se comunica

motivación

  • cuadros de diálogo en una interfaz gráfica
  • un cuadro de diálogo presenta una colección de componentes
  • a menudo de producen dependencias entre los componentes
  • diferentes diálogos presentan diferentes dependencias

solución

  • definir un objeto mediador independiente
  • responsable de controlar y coordinar las interacciones entre componentes
  • éstos sólo conocen al mediador y no necesitan tener constancia unos de otros

 consideraciones

  • un mediador reúne el comportamiento que, de otro modo, estaría distribuido entre varios objetos
    • extender dicho comportamiento sólo requiere heredar de la clase Mediador
  • un mediador permite el desacoplamiento entre componentes
    • reemplazando interacciones muchos-a-muchos por uno-a-muchos
  • los componentes han de comunicarse con su mediador cuando ocurre algo de interés
    • esto se puede implementar con ayuda del patrón Observador
  • no es necesario definir una clase abstracta Mediador cuando los componentes se organizan con un solo mediador

patrón Observador

  • patrón de comportamiento para objetos
  • define dependencias entre observacores y observables
    • si un observable cambia de estado todos los observadores son notificados automáticamente con independencia de cuántos puedan existir
  • es una forma de implementar el patrón MVC
    • proporciona mecanismos para registrar observadores en el objeto observable y notificar a los observadores cuando observable sufre alguna modificación
  • se aplica cuando:
    • se quiere mantener la consistencia entre objetos relacionados
    • sin acoplarlos fuertemente para no comprometer su reutilización

ejemplo

  • una GUI que ofrece diversas representaciones o vistas de los datos o modelo de una aplicación
  • las clases Observable (modelo) y Observadores (vistas) deben poder reutilizarse independientemente
  • las modificaciones al modelo deben repercutir en las vistas
    • incluso alguno de los diagramas puede ser editable, y modificaciones en él también repercutirían en el resto de diagramas y en el propio modelo

solución

 interacción

consideraciones

  • se aplica cuando:
    • una abstracción tiene más de un aspecto, unos dependientes de otros
    • se desea reutilizar cada uno de ellos independientemente de los otros
    • el cambio de un objeto requiere cambiar los otros, y no se sabe cuántos objetos deben actualizarse
    • un objeto debe notificar ciertos cambios a otros pero sin saber quiénes son dichos objetos
  • mínimo acoplamiento entre Observables y Observadores
    • basado en clases abstractas e interfaces
    • la notificación se realiza mediante difusión
    • un objeto Modelo no necesita saber quiénes son sus Observadores
  • si un mismo Observador tiene varios Observables
    • notificar() deberá pasar el objeto observado como argumento
  • cada cambio de estado dispara la notificación
    • puede producir un problema de actualizaciones en cascada
  • permite suscripción especializada
    • los observadores especifican qué cambios les son relevantes
  • mecanismo de notificación:
    • cada cambio de estado puede disparar una notificación: política simple, pero puede provocar cascadas de actualizaciones
    • el propio objeto observado decide qué cambios son relevantes para ser notificados
  • es conveniente evitar notificaiones específicas para observadores concretos, y llegar a un comprmiso entre:
    • un protocolo de notificación que envía a cada observador la información que necesita (a través de parámetros adicionales de la operación "actualizar"): alto acoplamiento entre observable y observadores, y baja reutilización
    • un protocolo de notificación que sólo advierte a los observacores de la existencia de cambios; cada observador debe ser capaz de saber qué información relevante es la que ha cambiado: ineficiencia
  • suscripción especializada de observadores:
    • al suscribir un observador a un observable puede ser interesante especificar qué tipo de aspectos pueden serle relevantes
    • en algunas librerías de clases (java.swing) se organizan jerarquías de esta tipo de aspectos (ActionListener)
  • actualizaciones en cascada cuando las relaciones entre observados y observadores son complejas:
    • varios objetos observados pueden estar estrechamente relacionados entre sí
    • las notificaciones a los observadores deben realizarse después de que los observables relacionados queden en un estado consistente
  • en estos casos, es conveniente complementar el patrón con un gestor de cambios cuya responsabilidad incluye:
    • mantener la asociación de cada observable con sus observadores, liberando de ello a los observables
    • definir una política de actualización
    • actualizar todos los observadores necesarioes a petición de un observable
    • el gestor puede especializarse con diferentes políticas, construyendo una jerarquía
